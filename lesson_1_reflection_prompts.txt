How did viewing a diff between two versions of a file help you see the bug that
was introduced?

    Diff помогает сравнивать файлы не просматривая полностью их содержимое. Он показывает только то чем отличаются два файла.

How could having easy access to the entire history of a file make you a more
efficient programmer in the long term?

    Ты можешь записывать свои изменения и не бояться что ты сделаешь что то не так, потому что всегда можно вернуться к предыдущей версии.

What do you think are the pros and cons of manually choosing when to create a
commit, like you do in Git, vs having versions automatically saved, like Google
docs does?

    Ты можешь выбирать когда делать коммит. И делать его тогда, когда логически это нужно делать. И не быть привязанным к конкретному промежутку времени или массиву информации.

Why do you think some version control systems, like Git, allow saving multiple
files in one commit, while others, like Google Docs, treat each file separately?

    Потому что некоторые программы состоят из нескольких файлов. В git можно их объединить в репозитории. И изменять сразу несколько файлов и подтверждать это все одним комитом. Так удобно делать изменения в больших массивах програмного кода в разных файла.

How can you use the commands git log and git diff to view the history of files?

    git log показывает нам все коммиты которые были сделаны во время работы над проектом. А git diff сравнивает любые два комита которые были сделаны. В итоге мы можем посмотреть изменения которые были добавлены в репозиторий или были удалены.

How might using version control make you more confident to make changes that
could break something?

    в контроле версий ты можешь сравнить две разные версии которые ты сохранил и вернуться к более ранней версии если ты совершил ошибку и исправить её, а сначала это облегчает процесс поиска ошибки!
